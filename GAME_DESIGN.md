# 토끼굴 수호대 - 게임 설계서

## 1. 목표

별토끼 사이트에 통합되는 전쟁시대 스타일의 모바일 우선 방어 미니게임이다.
유저의 누적 등급과 보유 코스튬을 게임 유닛에 연결하되, 상점 구매가 필수인
페이 투 윈 구조는 만들지 않는다.

- 총 스테이지: 20개
- 스테이지 제한 시간: 1분 30초
- 스테이지 사이 준비 시간: 10초
- 풀런 최대 시간: 33분 10초 (`20 * 90초 + 19 * 10초`)
- 하루 입장 횟수: 계정당 3회
- 일일 기준 시간대: 한국 시간(Asia/Seoul, KST)
- 인게임 자원: 당근
- 진행 방식: 매 실행마다 1스테이지부터 시작
- 지원 환경: 모바일 세로 화면 우선, 데스크톱 대응

## 2. 핵심 원칙

1. 클라이언트는 화면 표시와 입력 수집을 담당한다.
2. 입장 횟수 차감, 결과 판정, 스타 지급은 서버와 DB만 수행한다.
3. 클라이언트가 제출한 최종 스테이지나 처치 수를 그대로 신뢰하지 않는다.
4. 같은 요청이 여러 번 전송되어도 입장권과 보상이 중복 처리되지 않아야 한다.
5. 스테이지를 직접 잠그지는 않지만 전투 난이도로 성장 체감을 만든다. 신규 계정은
   5스테이지, 길토끼 운영은 10스테이지 근처, 별토끼 등급은 15스테이지 근처,
   별토끼와 코스튬 조합은 20스테이지 클리어를 목표로 한다.

## 3. 게임 루프

```text
입장 요청
  -> 서버가 입장권 차감, 실행 ID/시드/설정 버전 발급
  -> 당근 수급
  -> 유닛 생산
  -> 자동 전투
  -> 웨이브 전멸 시 다음 스테이지
  -> 스테이지 체크포인트 저장
  -> 게임오버 또는 20스테이지 클리어
  -> 서버 재검증
  -> 일일 최고 기록 및 미수령 마일스톤 보상 지급
```

종료 조건:

- 토끼굴 HP가 0이 되면 패배한다.
- 제한 시간 안에 웨이브를 처치하지 못하면 패배한다.
- 20스테이지 드래곤을 처치하면 승리한다.
- 브라우저 종료 또는 장시간 연결 중단은 포기 처리한다.

## 4. 화면 구성

```text
┌─────────────────────────────────────────┐
│ 토끼굴 HP: ████████░░ 80/100             │
│ 당근: 240     스테이지 3/20              │
│ 남은 시간: 1:12     오늘 도전: 2/3       │
├─────────────────────────────────────────┤
│ [전선]                                   │
│ 뱀 뱀 뱀          ->          토끼굴     │
│ 뱀 15/20 · 뱀 20/20 · 뱀 8/20           │
├─────────────────────────────────────────┤
│ [내 유닛]                                │
│ 길토끼 x3 · 들토끼 x1 · 6/10             │
├─────────────────────────────────────────┤
│ [유닛 생산 - 가로 스크롤]                │
│ [길 -10] [들 -30] [달 -80] [별 -200]    │
├─────────────────────────────────────────┤
│ [최근 전투 로그 3줄]                     │
│ > 뱀이 토끼굴 공격: -5 HP                │
│ > 길토끼가 뱀 처치: +15 당근             │
└─────────────────────────────────────────┘
```

- 생산 버튼은 하단 고정 영역에 배치한다.
- 잠긴 유닛은 잠금 조건을 표시하되 버튼을 숨기지 않는다.
- 전투 로그는 최근 3줄만 노출하고 전체 로그는 접을 수 있게 한다.
- HP, 당근, 제한 시간은 색상만으로 상태를 전달하지 않는다.
- `prefers-reduced-motion` 사용자는 이동 애니메이션을 최소화한다.

## 5. 전투 규칙

### 5.1 전선

- 전선은 `0~1000`의 논리 좌표를 사용한다.
- 토끼굴은 `x=950`, 적은 `x=0`에서 출현한다.
- 아군은 생산 순서대로 토끼굴 앞에 배치된다.
- 사거리 안에 적이 있으면 가장 앞선 적을 자동 공격한다.
- 적은 사거리 안에 아군이 있으면 아군을 공격하고, 없으면 토끼굴로 이동한다.
- 유닛과 적의 동시 공격은 같은 시뮬레이션 틱에서 함께 반영한다.

### 5.2 시뮬레이션

- 고정 틱: 100ms
- 서버 시작 시각과 클라이언트의 단조 증가 시계(`performance.now()`)를 함께 사용하고,
  체크포인트마다 서버 시각으로 오차를 보정한다.
- 공격 주기는 초 단위 간격이다. 예: `1.2초`는 1.2초마다 1회 공격한다.
- 동일 틱의 처리 순서: 생산 입력 -> 이동 -> 공격 예약 -> 피해 적용 -> 사망/보상 처리.
- 동일 대상 처치 보상은 한 번만 지급한다.
- 난수는 서버가 발급한 시드 기반 PRNG만 사용한다.
- 밸런스 수치는 `config_version`별 고정 설정으로 관리한다.

### 5.3 토끼굴

- 기본 HP: 100
- 스테이지 사이에 HP는 자동 회복되지 않는다.
- 준비 시간 중에도 유닛 생산은 가능하다.
- 향후 회복 효과가 추가되더라도 최대 HP를 넘을 수 없다.

### 5.4 유닛 제한

- 전장에 생존 가능한 아군은 최대 10마리다.
- 최대치에서는 생산 명령을 받을 수 없고 당근도 차감하지 않는다.
- 유닛이 사망하면 즉시 빈 슬롯이 생긴다.
- 실행 중 해금 상태는 시작 시점 스냅샷으로 고정한다.

## 6. 당근 자원

| 획득 방법 | 당근 |
|---|---:|
| 실행 시작 자원 | 30 |
| 시간 경과 10초마다 | 5 |
| 뱀 처치 | 15 |
| 여우 처치 | 25 |
| 늑대 처치 | 40 |
| 호랑이 처치 | 60 |
| 드래곤 처치 | 200 |

- 당근은 실행 안에서만 사용하며 다음 실행으로 이월되지 않는다.
- 시간 수급은 전투 중에만 발생하고 준비 시간에는 발생하지 않는다.
- 처치 당근은 서버 재검증 결과와 일치해야 한다.
- 당근은 음수가 될 수 없다.

## 7. 기본 유닛

현재 서비스의 등급은 `src/lib/points.ts`의 `POINT_LEVELS`를 단일 기준으로 사용한다.
DB에 별도 `tier` 문자열 컬럼을 만들지 않고 `profiles.earned_points`로 등급을 계산한다.

| 유닛 | 비용 | HP | 공격력 | 공격 주기 | 사거리 | 해금 조건 |
|---|---:|---:|---:|---:|---:|---|
| 길토끼 | 10 | 20 | 5 | 1.0초 | 75 | 기본 제공 |
| 풀토끼 | 20 | 30 | 8 | 1.0초 | 80 | 풀토끼 이상 |
| 들토끼 | 35 | 48 | 13 | 1.2초 | 85 | 들토끼 이상 |
| 달토끼 | 80 | 90 | 25 | 1.5초 | 120 | 달토끼 이상 |
| 별토끼 | 170 | 170 | 48 | 1.8초 | 130 | 별토끼 이상 |
| 은하토끼 | 250 | 210 | 65 | 1.6초 | 150 | 은하토끼 이상 |
| 우주토끼 | 340 | 280 | 88 | 1.8초 | 165 | 우주토끼 이상 |
| 전설토끼 | 450 | 360 | 120 | 2.0초 | 180 | 전설토끼 |

- 높은 등급 유닛은 강하지만 비용 대비 효율이 항상 상위 호환이 되지는 않게 조정한다.
- 낮은 비용 유닛은 몸막기와 빠른 충원 용도로 20스테이지까지 의미가 있어야 한다.
- 표의 수치는 1차 밸런스 초안이며 플레이테스트 후 `config_version`을 올려 변경한다.

## 8. 상점 코스튬 유닛

기존 `shop_items.type = 'costume'`과 `user_items` 보유 여부를 사용한다.
코스튬은 별도 최상위 유닛이 아니라 기본 유닛의 사이드그레이드로 제공한다.

| 코스튬 | 역할 예시 | 원칙 |
|---|---|---|
| 닌자토끼 | 낮은 HP, 빠른 공격 | 단일 대상 특화 |
| 무사토끼 | 방어와 반격 | 안정형 |
| 공주토끼 | 주변 아군 소폭 강화 | 지원형 |
| 마법사토끼 | 느린 범위 공격 | 다수전 특화 |
| 해적토끼 | 처치 당근 보너스 | 자원형 |
| 우주비행사토끼 | 긴 사거리 | 후방형 |
| 악마토끼 | HP를 소모해 공격 강화 | 고위험형 |
| 산타토끼 | 주기적 소량 회복 | 유지형 |

- 보유한 코스튬만 선택할 수 있다.
- 장착 여부와 무관하게 보유 중이면 해금한다.
- 기간제 아이템은 실행 시작 시 만료되지 않은 경우에만 해금한다.
- 코스튬별 정확한 수치는 별도 밸런스 설정에 둔다.
- 코스튬이 없어도 모든 스테이지를 클리어할 수 있어야 한다.

## 9. 적과 웨이브

| 적 | 기본 HP | 공격력 | 공격 주기 | 이동 속도 | 처치 당근 |
|---|---:|---:|---:|---:|---:|
| 뱀 | 20 | 5 | 1.5초 | 45 | 15 |
| 여우 | 45 | 10 | 1.2초 | 70 | 25 |
| 늑대 | 85 | 18 | 1.5초 | 50 | 40 |
| 호랑이 | 160 | 32 | 2.0초 | 35 | 60 |
| 드래곤 | 2000 | 55 | 2.5초 | 20 | 200 |

| 스테이지 | 구성 | 수량 범위 | 추가 규칙 |
|---|---|---:|---|
| 1~4 | 뱀 | 5~10 | 기본 학습 구간 |
| 5~9 | 뱀 + 여우 | 8~12 | 빠른 적 등장 |
| 10~14 | 여우 + 늑대 | 8~12 | 전열 유지 필요 |
| 15~19 | 늑대 + 호랑이 | 6~10 | 고HP 적 중심 |
| 20 | 드래곤 | 1 | 보스전 |

- 웨이브 구성과 출현 시점은 서버 시드와 설정 버전으로 결정한다.
- 드래곤에게 특정 등급 미만 공격을 무효화하는 하드 방어막은 두지 않는다.
- 대신 드래곤은 전투 시작 후 20초 동안 받는 피해 30% 감소 효과를 가진다.
- 모든 등급이 클리어 가능하되, 낮은 등급은 자원 운영과 유닛 조합 난도가 높다.

## 10. 스테이지 진행

- 각 스테이지는 웨이브 1개이며 제한 시간은 90초다.
- 적을 전멸시키면 남은 시간과 관계없이 스테이지를 종료한다.
- 1~19스테이지 종료 후 10초 준비 시간을 제공한다.
- 준비 시간이 끝나면 자동으로 다음 스테이지를 시작한다.
- 토끼굴 HP와 당근은 다음 스테이지로 이어진다.
- 생존 유닛은 스테이지 클리어 시 전원 귀환하며 다음 스테이지는 새로 편성한다.
- 클리어 시 `80 + (클리어 스테이지 * 35)` 당근을 다음 작전 보급으로 지급한다.
- 제한 시간 만료 시 해당 틱까지 예약된 피해를 적용한 뒤 승패를 판정한다.
- 마지막 적 처치와 토끼굴 파괴가 같은 틱이면 클리어를 우선한다.

## 11. 일일 입장 정책

- KST 날짜별 계정당 최대 3회 입장할 수 있다.
- 입장 횟수는 `start_game_run` DB 함수가 새 실행을 만들 때 원자적으로 증가시킨다.
- 실행 페이지를 열기만 한 것은 입장으로 계산하지 않는다.
- 실행 ID와 시드 발급이 성공한 시점부터 입장 1회를 소비한다.
- 사용자가 즉시 나가거나 새로고침해도 입장 횟수는 복구하지 않는다.
- 동시에 여러 탭에서 시작 요청이 와도 3회를 초과할 수 없다.
- 진행 중인 실행은 계정당 하나만 허용한다.
- 정상 진행 중 새로고침하면 마지막 체크포인트부터 한 번만 복구할 수 있다.
- 마지막 체크포인트 이후 최대 1개 스테이지의 진행은 유실될 수 있다.
- 실행 시작 후 45분이 지나면 만료 처리한다.

## 12. 포인트 보상

서비스 용어에 맞춰 보상 단위는 `스타`로 표기한다.
보상은 매 실행에서 클리어한 스테이지당 10스타를 지급한다. 각 실행은 항상
1스테이지부터 시작하며 같은 실행의 중복 종료 요청만 멱등 처리한다.

- 한 판에 8스테이지를 클리어하면 `8 * 10 = 80스타`를 지급한다.
- 다음 도전에서 12스테이지를 클리어하면 해당 판의 120스타를 다시 지급한다.
- 하루 3회, 판당 최대 200스타이므로 하루 최대 게임 보상은 600스타다.
- unique key는 `game:{run_id}:stage:{stage}` 형식으로 스테이지마다 기록한다.
- 20스테이지 드래곤 보상은 20스테이지의 10스타에 포함하며 별도 중복 지급하지 않는다.
- 스타 원장 기록과 프로필 갱신은 기존 `award_points`의 멱등 처리 방식을 재사용하되,
  클라이언트가 범용 지급 RPC를 직접 호출할 수 없게 권한을 먼저 정리한다.
- unique key 형식은
  `game:{user_id}:{kst_date}:{milestone}`으로 고정한다.
- 지급된 스타는 `points`와 `earned_points`에 모두 반영된다.
- 밸런스 변경 전 최소 1주간 평균 도달 스테이지와 일일 지급량을 관찰한다.

## 13. 서버 검증과 부정행위 방지

### 13.1 시작

서버의 `start_game_run` 처리는 다음을 한 트랜잭션에서 수행한다.

1. 로그인 사용자 확인
2. KST 기준 일일 행 잠금 또는 생성
3. 남은 입장 횟수 확인 및 증가
4. 진행 중인 기존 실행 만료 여부 확인
5. 암호학적으로 안전한 시드 생성
6. 현재 `config_version`, 등급, 보유 코스튬 스냅샷 저장
7. `run_id` 반환

### 13.2 진행

클라이언트는 결과 수치가 아니라 다음 명령 로그를 제출한다.

```ts
type GameCommand = {
  seq: number;
  elapsedMs: number;
  type: 'spawn';
  unitKey: string;
};
```

- `seq`는 1부터 연속 증가해야 한다.
- `elapsedMs`는 이전 명령보다 작을 수 없다.
- 서버는 시드, 설정, 명령 로그로 같은 전투를 재현한다.
- 생산 시점의 당근, 유닛 슬롯, 해금 조건을 서버가 다시 검사한다.
- 제출된 `elapsedMs`는 실행의 실제 서버 경과 시간보다 앞설 수 없다. 네트워크 오차는
  최대 5초만 허용해 30분 분량의 실행을 즉시 완료하는 요청을 거절한다.
- 체크포인트는 스테이지 종료 시 제출하며 DB에는 최대 20개만 저장한다.
- 로그 크기와 요청 본문 크기에 상한을 둔다.

### 13.3 종료

- 서버 재현 결과만 최종 스테이지, 승패, 처치 수로 저장한다.
- 클라이언트 결과와 서버 결과가 다르면 보상을 지급하지 않고 실행을 `rejected`로 종료한다.
- `finish_game_run`은 이미 종료된 실행에 다시 호출되어도 기존 결과를 반환한다.
- 보상 지급과 실행 종료는 하나의 DB 트랜잭션에서 처리한다.
- 서버 액션에서 service role로 임의 포인트를 직접 수정하지 않는다.
- `finish_game_run`이 내부 전용 포인트 함수를 호출하고, 브라우저는 해당 함수를
  직접 실행할 권한을 갖지 않는다.
- unique key 충돌은 이미 지급된 정상 요청으로 처리하고 포인트를 다시 더하지 않는다.

## 14. 탭 비활성화와 연결 중단

- `setInterval` 호출 횟수를 게임 시간으로 사용하지 않는다.
- 화면이 숨겨져도 서버 기준 경과 시간은 계속 흐른다.
- 복귀 시 누락된 시간을 고정 틱 단위로 따라잡되 한 프레임에서 최대 처리량을 제한한다.
- 클라이언트 시간이 서버 시간과 5초 이상 어긋나면 체크포인트를 다시 동기화한다.
- 오프라인 상태가 30초를 넘으면 입력을 막고 재연결을 시도한다.
- 2분 이상 재연결하지 못하면 마지막 체크포인트 기준으로 포기 처리한다.

## 15. 데이터 모델

`game_daily_stats`는 날짜별 제한과 최고 기록을, `game_runs`는 개별 실행을,
`game_run_checkpoints`는 검증 가능한 진행 상태를 저장한다.

```sql
create table public.game_daily_stats (
  user_id uuid not null references public.profiles(id) on delete cascade,
  game_date date not null,
  attempts_used smallint not null default 0
    check (attempts_used between 0 and 3),
  best_stage smallint not null default 0
    check (best_stage between 0 and 20),
  dragon_defeated boolean not null default false,
  updated_at timestamptz not null default now(),
  primary key (user_id, game_date)
);

create type public.game_run_status as enum (
  'active', 'cleared', 'failed', 'abandoned', 'expired', 'rejected'
);

create table public.game_runs (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references public.profiles(id) on delete cascade,
  game_date date not null,
  attempt_no smallint not null check (attempt_no between 1 and 3),
  status public.game_run_status not null default 'active',
  seed bigint not null,
  config_version int not null,
  unlock_snapshot jsonb not null default '{}'::jsonb,
  final_stage smallint not null default 0
    check (final_stage between 0 and 20),
  dragon_defeated boolean not null default false,
  started_at timestamptz not null default now(),
  finished_at timestamptz,
  expires_at timestamptz not null,
  unique (user_id, game_date, attempt_no)
);

create unique index game_runs_one_active_per_user_idx
  on public.game_runs (user_id)
  where status = 'active';

create table public.game_run_checkpoints (
  run_id uuid not null references public.game_runs(id) on delete cascade,
  stage smallint not null check (stage between 1 and 20),
  command_log jsonb not null,
  state_hash text not null,
  elapsed_ms int not null check (elapsed_ms >= 0),
  created_at timestamptz not null default now(),
  primary key (run_id, stage)
);

create index game_runs_user_started_at_idx
  on public.game_runs (user_id, started_at desc);
```

추가 DB 규칙:

- `game_daily_stats.user_id`와 `game_runs.user_id`는 `public.profiles(id)`를 참조한다.
  신규 가입자가 profiles 행 없이 게임에 접근하면 FK 오류가 발생한다.
  `start_game_run` 함수 진입 시 `profiles` 행 존재 여부를 먼저 확인하고,
  없으면 사용자에게 프로필 설정 안내를 반환한다.
  (`supabase/fix-auth-user-trigger.sql`의 트리거가 정상 작동하면 일반적으로 발생하지 않으나
  트리거 누락 계정에 대한 방어 처리가 필요하다.)
- `game_date`는 클라이언트가 보내지 않고 DB 함수가 KST로 계산한다.
- 계산식은 `(now() at time zone 'Asia/Seoul')::date`를 사용한다.
- 클라이언트는 게임 테이블을 직접 수정할 수 없다.
- RLS에서 본인 실행의 읽기만 허용하고 쓰기는 SECURITY DEFINER RPC로 제한한다.
- `start_game_run`, `save_game_checkpoint`, `finish_game_run` 함수의
  `search_path`는 `public`으로 고정하고 `auth.uid()`를 검증한다.
- `state_hash`는 클라이언트 값을 저장하지 않고 서버 재현 결과로 계산한다.
- 실행별 최고 기록과 역대 최고 기록은 조회 시 집계하거나 별도 통계 뷰로 제공한다.

## 16. 기존 시스템 연동

### 등급

- `profiles.earned_points`를 조회한다.
- `src/lib/points.ts`의 `getPointLevel()`과 `POINT_LEVELS`가 표시 및 해금 기준이다.
- 서버 검증용 SQL/TypeScript 등급 키는 한 곳에서 생성해 중복 정의를 피한다.

### 상점

- `shop_items.costume_key`와 `user_items`를 사용한다.
- 실행 시작 시 보유 아이템과 `expires_at`을 확인해 스냅샷에 저장한다.
- 실행 도중 구매, 장착, 만료된 아이템은 현재 실행에 영향을 주지 않는다.

### 스타 지급

- `supabase/add-auto-attend-and-replies.sql` 마지막 줄에
  `GRANT EXECUTE ON FUNCTION public.award_points(...) TO authenticated;`가 있다.
  이 권한이 살아 있으면 브라우저 콘솔에서 임의 금액을 직접 지급할 수 있으므로
  게임 출시 전에 반드시 제거해야 한다.
  ```sql
  REVOKE EXECUTE ON FUNCTION public.award_points(uuid, int, text, text, jsonb)
    FROM authenticated;
  ```
- 권한 제거 후 아래 호출부가 깨지므로 service role 클라이언트로 교체해야 한다.
  - `src/app/cheer/actions.ts` — 응원전 보상
  - `src/app/api/reply/route.ts` — 댓글 보상
  - `src/app/actions.ts` — 리뷰/기타 보상 (서버 액션이지만 anon 클라이언트 사용 여부 확인)
- 출석 체크(`src/lib/points.ts`)와 어드민(`src/app/admin/actions.ts`)은
  이미 서버 전용 경로이므로 별도 조치 불필요.
- 게임 보상은 `finish_game_run` 트랜잭션 내부에서만 포인트 원장과 프로필을 갱신한다.
- reason 예시: `토끼굴 수호대 10스테이지 보상`.
- metadata에는 `run_id`, `game_date`, `milestone`, `config_version`을 기록한다.

## 18. 테스트 기준

### 엔진 단위 테스트

- 같은 시드와 명령은 항상 같은 결과를 만든다.
- 당근 부족, 슬롯 초과, 잠긴 유닛 생산이 거절된다.
- 동시 처치 보상이 중복되지 않는다.
- 제한 시간과 동시 발생한 처치의 승패 규칙이 유지된다.
- 드래곤 피해 감소가 정확히 종료된다.

### DB/RPC 테스트

- 네 개의 동시 시작 요청에서도 최대 3개만 성공한다.
- 같은 종료 요청을 반복해도 보상은 한 번만 지급된다.
- 실제 실행 시간보다 앞선 `elapsedMs`로 제출한 체크포인트와 종료 요청이 거절된다.
- 같은 날 더 높은 마일스톤 달성 시 차액만 지급된다.
- KST 자정 전후 입장 횟수가 올바른 날짜에 기록된다.
- 다른 사용자의 실행을 읽거나 수정할 수 없다.
- 만료된 코스튬이 해금 스냅샷에 포함되지 않는다.

### UI 테스트

- 360px 너비에서 생산과 상태 확인이 가능하다.
- 백그라운드 전환 후 전투 상태가 서버와 다시 일치한다.
- 새로고침 후 마지막 체크포인트를 복구한다.
- 로그인 만료, 네트워크 끊김, 입장 횟수 소진 오류를 구분해 표시한다.

## 19. 운영 지표

- 일일 게임 참여 사용자 수
- 사용자당 평균 시도 횟수
- 스테이지별 도달률과 실패 원인
- 유닛별 생산률과 생존 시간
- 등급별 20스테이지 클리어율
- 평균 실행 시간과 중도 이탈 스테이지
- 일일 평균/최대 게임 스타 지급량
- 서버 재검증 실패 및 `rejected` 비율

## 21. 출시 완료 조건

- 클라이언트 값 변조만으로 스타를 획득할 수 없다.
- 범용 `award_points`를 브라우저에서 임의 호출할 수 없다.

- 새로고침과 일시적 네트워크 중단에서 정의된 방식으로 복구된다.
- 엔진, RPC, 주요 UI 흐름 테스트가 통과한다.
- 게임 보상으로 기존 스타 경제가 급격히 변하지 않는지 운영 검토가 끝난다.
